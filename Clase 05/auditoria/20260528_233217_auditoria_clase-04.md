## 🔴 PROBLEMAS CRÍTICOS (Arreglar inmediatamente)

### 1. Bloqueo de Ejecuciones mediante `it.only`
* **Dónde:** 
  * `cypress/e2e/carrito.cy.js` (Línea 33): `it.only('T12 Eliminar un producto...')`
  * `cypress/e2e/Checkout.cy.js` (Línea 23): `it.only('T14 Completar checkout con error')`
* **Por qué es crítico:** El uso de `.only` restringe la ejecución de la suite a un solo caso de prueba, ignorando los demás tests. Esto es inaceptable en flujos de integración continua (CI/CD) y previene la validación global de regresiones.
* **Fix:** Remover `.only` para permitir la ejecución de la suite completa.
  ```javascript
  // Antes
  it.only('T12 Eliminar un producto...', () => { ... })
  
  // Después
  it('T12 Eliminar un producto...', () => { ... })
  ```

### 2. Duplicación Masiva e Inicio de Sesión Hardcodeado
* **Dónde:** `beforeEach` en `carrito.cy.js`, `Checkout.cy.js`, e `inventario.cy.js`.
* **Por qué es crítico:** Cada archivo de prueba repite exactamente el mismo bloque de inicio de sesión (`cy.visit()`, `.type('standard_user')`, `.type('secret_sauce')`, `.click()`). Cualquier cambio en el formulario de inicio de sesión o las credenciales romperá todas las suites de prueba, exigiendo mantenimiento manual en múltiples archivos (violación del principio DRY).
* **Fix:** Encapsular la lógica de login en un comando personalizado en `cypress/support/commands.js` o en un Page Object.

### 3. Credenciales y URLs Sensibles en el Código Fuente
* **Dónde:** `beforeEach` en todos los archivos.
* **Por qué es crítico:** Hardcodear credenciales reales (`standard_user`, `secret_sauce`) y URLs de producción/prueba directamente en el código de las pruebas representa un grave riesgo de seguridad si el repositorio es público, y limita la flexibilidad de ejecutar los tests en múltiples ambientes (QA, Staging, Dev).
* **Fix:** Trasladar la URL a `cypress.config.js` (`baseUrl`) y las credenciales a variables de entorno (`Cypress.env`).

## ⚠️ MEJORAS IMPORTANTES (Arreglar pronto)

### 1. Ausencia del Patrón de Diseño Page Object Model (POM)
* **Dónde:** Todos los archivos en `cypress/e2e/`.
* **Por qué es importante:** Los tests interactúan directamente con los selectores de la página web. Si la interfaz cambia, se deben editar docenas de líneas de código en múltiples archivos. POM centraliza los elementos de la interfaz en clases dedicadas, de modo que los cambios en la UI solo se actualizan en un único lugar.
* **Fix:** Implementar clases de página para Login, Inventario, Carrito y Checkout.

### 2. Uso de Selectores de Clases CSS Frágiles
* **Dónde:**
  * `Checkout.cy.js` (Líneas 28-29) y `login.cy.js` (Líneas 18, 26, 35): `.error-message-container`
  * `inventario.cy.js` (Líneas 19-20): `.inventory_item_name`, `.inventory_item_price`
  * `login.cy.js` (Línea 43): `.bm-burger-button`
* **Por qué es importante:** Las clases CSS están diseñadas para fines de estilo y diseño. Son altamente volátiles y cambian con frecuencia durante las refactorizaciones del frontend. El framework ya cuenta con atributos `data-test` robustos en la mayoría de los elementos, por lo que su uso debe priorizarse siempre.
* **Fix:** Utilizar selectores basados en atributos de prueba dedicados.
  ```javascript
  // Antes
  cy.get('.error-message-container').should('be.visible')
  
  // Después
  cy.get('[data-test="error"]').should('be.visible')
  ```

### 3. Falta de Uniformidad y Consistencia
* **Dónde:** Estructura general de nombres de archivos y tests.
* **Por qué es importante:** 
  * Inconsistencia en nomenclatura de archivos: `Checkout.cy.js` (con mayúscula inicial) vs. `carrito.cy.js` (minúscula).
  * Inconsistencia en títulos de tests: Algunos usan ID de caso de prueba (`T10`, `T11`, `T4`, `T5`) mientras que otros usan nombres planos descriptivos.
* **Fix:** Establecer y aplicar una guía de estilos homogénea (ej: todos los archivos en minúsculas tipo `checkout.cy.js` y nombres de tests estandarizados).

### 2. Implementación de Page Object de Ejemplo: `cypress/pages/LoginPage.js`
```javascript
class LoginPage {
  // Localizadores encapsulados
  get usernameInput() { return cy.get('[data-test="username"]'); }
  get passwordInput() { return cy.get('[data-test="password"]'); }
  get loginButton() { return cy.get('[data-test="login-button"]'); }
  get errorMessage() { return cy.get('[data-test="error"]'); }

  // Métodos de acción
  visit() {
    cy.visit('/');
  }

  login(username, password) {
    if (username) this.usernameInput.type(username);
    if (password) this.passwordInput.type(password);
    this.loginButton.click();
  }
}

export default new LoginPage();
```

### 3. Refactorización del Test usando POM: `cypress/e2e/login.cy.js`
```javascript

describe('Suite de Login - SauceDemo', () => {
  beforeEach(() => {
    LoginPage.visit();
  });

  it('TC-01: Debe permitir el login exitoso con credenciales estándar', () => {

    cy.url().should('include', '/inventory.html');

  it('TC-02: Debe mostrar error con credenciales incorrectas', () => {

    LoginPage.errorMessage
      .should('be.visible')
      .and('contain.text', 'Epic sadface: Username and password do not match');
  });
});
```
